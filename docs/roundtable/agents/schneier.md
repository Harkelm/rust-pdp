### Bruce Schneier -- Security Expert
> "You have a formally verified engine sitting behind an informally trusted trust boundary -- that's where attackers will go."

#### Findings

| # | Finding | Severity | File:Line | Detail |
|---|---------|----------|-----------|--------|
| 1 | Fail-open is a configuration option, not a hard prohibition | P0 | `kong-plugin-architecture.md:68` | `FailOpen bool` is a config field with no documented default. The PDP doc says "never fail-open unless explicitly configured" but the plugin exposes it as a user-facing toggle. Any misconfiguration, deployment automation error, or operator mistake silently converts a deny-by-default system into an allow-by-default system. There is no warning, no audit log entry, no circuit breaker around this flag. |
| 2 | JWT claims are PEP-supplied and the trust boundary is underdefined | P0 | `rust-pdp-service-architecture.md:152-158`, `entitlement-translation.md:119-148` | The architecture says "PDP should NOT trust entity data from PEP for security-critical attributes" but the JWT-to-Cedar mapping in `entitlement-translation.md` constructs principal entities -- including role memberships -- directly from JWT claims. Role membership IS a security-critical attribute. The claim "JWT already validated" conflates authentication (signature valid) with authorization (claims are accurate and not stale). A compromised token issuer, a misconfigured scope, or a post-issuance revocation all produce valid-signature tokens with wrong entitlements. |
| 3 | Admin reload endpoint is unauthenticated in the spec | P1 | `rust-pdp-service-architecture.md:130` | `POST /admin/reload` is listed as a reload trigger with no authentication requirement shown. If an attacker can reach this endpoint -- and in a sidecar deployment, anything on localhost can -- they can force continuous policy reloads (DoS via CPU), attempt to race a malicious policy file into the window between file write and validation, or exhaust the entity cache by forcing repeated invalidation. The validation-before-swap protects against broken policies but not against an attacker who controls the file system AND the reload trigger. |
| 4 | Decision cache can preserve stale allows past revocation | P1 | `rust-pdp-service-architecture.md:163-179` | The cache key is `principal:action:resource` with a 60s TTL. The docs note "never cache forbid decisions" but the inverse problem is worse: cached allow decisions survive policy changes, entity revocations, and group membership removals for up to TTL seconds. For a revoked API key or a fired employee, 60 seconds of continued access is a real window. Policy-change-triggered invalidation is listed but not mandated. |
| 5 | Skip-on-error in Cedar evaluation silently narrows policy coverage | P1 | `cedar-policy-language.md:50-51` | Cedar's skip-on-error semantics: "policies that error during evaluation are skipped, not treated as deny." This is documented as preventing global lockout, but it has an attacker-exploitable consequence: if a forbid policy errors (e.g., missing entity attribute), it is skipped, and a permit policy can fire. The decision is Allow when it should be Deny. An attacker who can induce evaluation errors on specific forbid policies -- by supplying malformed entity data -- can potentially bypass deny rules. |
| 6 | HTTP vs gRPC security posture gap is not threat-modeled | P2 | `kong-plugin-architecture.md:170-178` | The decision matrix lists Go+HTTP as the starting point with no threat analysis. HTTP/JSON to the PDP means the request body is parsed twice (JSON decode in Go plugin, JSON decode in Rust PDP) -- two attack surfaces. gRPC with protobuf reduces parsing attack surface and enables mTLS more naturally. The wiki notes "gRPC over HTTP: lower overhead" but frames this as a performance decision, not a security decision. |

#### Assessment

The Cedar engine itself is the strongest part of this architecture. Formally verified, default-deny, bounded evaluation -- these are properties you want in a security-critical system. The Rust implementation choice adds memory safety. The problem is that sound components can be assembled into an insecure system, and that's what I see here.

The most dangerous finding is the fail-open configuration option (`kong-plugin-architecture.md:68`). The PDP documentation says "never fail-open unless explicitly configured" -- but "explicitly configured" means an operator can configure it. In practice, availability pressure at 3am will produce fail-open deployments. The architecture should treat fail-open as an emergency override with mandatory audit logging, a time limit, and a pager alert, not as a routine config flag.

The JWT trust boundary finding (`entitlement-translation.md:129-148`) is subtler but more structurally concerning. The architecture correctly states the PDP should not trust PEP-supplied security-critical attributes, then immediately shows a production-tested pattern where the PDP constructs principal entities including role memberships from JWT claims. This is not a theoretical concern: JWT claims can be stale (issued before revocation), wrong (misconfigured issuer), or forged (compromised signing key). The PDP should independently resolve role memberships for security-sensitive decisions, using JWT claims only as hints to look up authoritative data.

The skip-on-error behavior (`cedar-policy-language.md:50-51`) deserves a threat model entry it does not currently have. If an attacker can induce evaluation errors on forbid policies -- by crafting requests with missing entity attributes that the forbid policy requires -- they can cause the deny to be skipped while a permit fires. This is a real injection vector against the policy evaluation engine, and the mitigation is not obvious: strict schema validation on all entity attributes before evaluation would help, but the architecture does not mandate this.

Before this ships, I want to see: (1) a written threat model naming adversaries and their capabilities, (2) the trust boundary between PEP-supplied and PDP-resolved attributes spelled out as a list of which attributes are resolved where, (3) the admin reload endpoint authenticated and rate-limited, (4) fail-open restricted to a time-limited emergency mode with mandatory alerting, and (5) a policy-change-triggered cache invalidation as the mandatory path, not an option.

#### Deliberation Addendum

**Positions revised**:
- F1 (fail-open): Upgraded from P0 with vague prescription to P0 with specific ceremony requirements. A secure fail-open override requires: mandatory audit log on activation (who, when, which routes, incident reference), security team alerting on activation, automatic time-bound expiry, and per-route scope only. Global fail-open is equivalent to disabling authorization. DHH's complementary fix is also correct: non-blocking routes should be modeled as Cedar `permit` policies, making authorization intent visible and auditable rather than hiding it in a config boolean.
- F2 (JWT entity trust): Framing revised per Lamport's sharpening. The architecture names the problem but does not specify the mechanism. "Check a real-time deny list in context" (`entitlement-translation.md:176-179`) specifies none of: who maintains the list, who checks it (plugin, PDP, or entity resolver), latency bounds, or what happens when the deny list is unavailable. A revoked credential that can't be confirmed revoked because the deny list is unreachable should default-deny -- the architecture is silent on this. Still P0, now grounded in specification gap.
- F3 (skip-on-error): Severity revised P1 -> P2 following Lamport's Round 3 confirmation. `Entities::from_entities` with `Some(&schema)` fails at entity construction time if a required attribute is absent -- before evaluation, before skip-on-error can fire. This closes the attack for required attributes. However, the mitigation is a policy authoring discipline, not an engine guarantee: all security-critical attributes referenced in forbid policy conditions must be marked required (not `?`) in the Cedar schema. Optional attributes remain exploitable. The architecture must specify a convention that forbid conditions may only reference required schema attributes -- this rule is not currently documented anywhere.

**Positions reinforced**:
- F1 (fail-open): Unanimous P0. Cantrill, Fowler, Hightower, Lamport, and DHH all independently reached the same conclusion. No dissent.
- F2 (entity trust, entity partition): Fowler proposed trusted/untrusted partition of entities array as structural fix. Held that partition alone is insufficient without enforcement invariant: forbid conditions must reference PDP-resolved attributes only, never PEP-supplied. Fowler accepted this as a refinement. Combined fix: partition as API contract + invariant as code-reviewable rule. Both are necessary.
- Cedar over OPA: DHH challenged Cedar choice on ecosystem grounds. Held position. Cedar's formal verification (Lean proofs, SMT analysis tools), non-Turing-completeness (bounded evaluation complexity, no DoS via policy construction), and structural default-deny are security properties OPA cannot match. Ecosystem operational cost is real but is the correct trade-off for a security-critical authorization path. OPA's existing Kong plugin means trusting unaudited third-party integration code in the critical path.
- Admin reload endpoint unauthenticated (F3 original): Not challenged by any panelist. Stands at P1.

**New observations**:
- Fowler identified the AuthZen `subject.properties` field (`rust-pdp-service-architecture.md:70-74`) as a freeform JSON bag with no schema constraint shown anywhere in the architecture. The mapping function from AuthZen properties to Cedar entity attributes is unspecified. If properties flow directly into Cedar entity attributes, the external-facing endpoint has a self-elevation attack surface: attacker POSTs `subject.properties: {"clearance_level": 5, "roles": ["admin"]}` and the mapping function elevates them. This is P0 on the external API surface -- worse than the Cedar-native endpoint because the standards-compliant API obscures the injection vector.
- Lamport identified that the cache key `principal:action:resource` (`rust-pdp-service-architecture.md:168-170`) has no policy version component. A new forbid policy does not invalidate matching cached allows -- it relies entirely on the invalidation mechanism firing correctly and in order. A policy generation counter in the cache key would make correctness independent of invalidation timing. This compounds F4.
- AGI-Accelerationist raised a finding that sharpens the fail-open concern: at scale, fail-closed under PDP overload means simultaneous denial across all requests -- operationally indistinguishable from a DDoS. The architecture must distinguish PDP-says-deny (403) from PDP-is-unavailable (503+Retry-After). The dangerous failure mode is timeout interpreted as security denial, which masks an availability problem as a security incident. Correct protocol: PDP 503 -> plugin 503+Retry-After; PDP decision=Deny -> plugin 403; PDP timeout -> plugin 503 (not 403). Fail-open ceremony remains P0; this adds a parallel P1: the plugin must implement explicit response code differentiation, which the architecture does not currently specify.

**Score changes**:
- Security: 0.45 -> 0.38. The AuthZen `subject.properties` injection surface is an additional P0 not in Phase 1. The unspecified deny-list failure behavior compounds F2. The architecture has more unspecified security invariants than initially counted. Skip-on-error revised to P2 (partially mitigated by schema validation, residual risk on optional attributes).
- Completeness: 0.55 -> 0.40. Lamport's specification gap distinction is correct: naming a security problem without specifying the invariant, mechanism, and failure behavior is not a solution. The AuthZen mapping function, deny-list ownership and failure behavior, entity attribute classification, admin endpoint authentication, plugin response code differentiation (503 vs 403), and forbid-on-required-attributes convention are all unspecified. The final round revealed more gaps than Round 2.
