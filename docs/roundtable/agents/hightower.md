#### Assessment

The architecture is technically sound. Cedar is the right engine -- sub-ms evaluation, Rust-native, formally verified, default deny. The observability section is better than most early-stage designs. The API shape (AuthZen for external, Cedar-native for internal) is a sensible boundary.

But the deployment topology recommendation (Go plugin + sidecar) optimizes for development speed at the cost of operational clarity. The Go plugin adds IPC overhead that cannot be eliminated without rewriting the plugin in Lua. Starting there and migrating later is exactly the kind of "we'll fix it when it matters" reasoning that creates permanent technical debt. The benchmark numbers are in the document -- 25% throughput reduction is the floor. That number should drive the plugin choice, not development convenience.

The sidecar recommendation conflates two independent concerns: low latency (solved by localhost networking) and lifecycle (introduced as a side effect). A remote PDP with connection pooling and a local decision cache in the plugin achieves localhost-comparable latency for the vast majority of cached decisions while keeping PDP independently deployable. The 30-60s TTL cache in the plugin means most requests never leave the plugin at all.

The entity resolution pattern is the most operationally fragile piece. Three data sources per request (policy file, entity cache, EntityDB) means three things that can fail independently. The document treats DB as a background concern but it is a foreground dependency. The readiness probe should check DB connectivity. The metrics should include DB query p99. The circuit breaker on DB failure should be explicit, not assumed.

What I'd want to see before this ships: (1) A deployment diagram that shows exactly what runs in each pod and what restarts what. (2) An on-call runbook: "authz returning 403 unexpectedly -- check X, then Y, then Z." (3) A decision on whether fail-open is per-route configurable or a deployment-level constant. (4) Lua plugin benchmarks against Go plugin at the target request rate -- not to mandate Lua, but to make the tradeoff explicit before the choice hardens.

#### Deliberation Addendum

**Positions revised**:
- **Sidecar coupling (Finding 2) -- concern reframed**: Fowler's cache-as-decoupling argument is correct. A plugin-side decision cache with TTL means PDP unavailability (including pod restart for policy update) is invisible to most traffic. My original framing conflated lifecycle coupling with latency. Revised position: the requirement is "plugin must have a decision cache before any topology choice is made." Topology is secondary to that prerequisite. Deployment complexity score revised upward (see below).
- **Go IPC tax (Finding 1) -- prescription sharpened**: Cantrill pushed back that Lua isn't obviously right -- Lua's weaker tooling and Kong sandbox pain are real costs. Revised position: neither Go nor Lua is universally correct. The architecture must state the latency budget explicitly, then derive the plugin language from it. The finding stands as P1 but the prescription changes from "reconsider Go" to "document the latency budget and derive the language choice from it."

**Positions reinforced**:
- **Fail-open as trapdoor (Finding 3) -- P0, unanimous**: Fowler, Cantrill, Schneier, DHH, and I all flagged `FailOpen bool` (`kong-plugin-architecture.md:68`) independently across separate analyses. No panelist defended it. This is a required architecture constraint: fail-open must not be a per-route runtime config toggle. If an emergency override is needed it requires a deployment change with mandatory audit logging and an automatic expiry timer.
- **DB as hidden first-class dependency (Finding 4) -- reinforced**: Schneier's JWT role freshness argument adds a required DB call for security-critical role resolution. Hashimoto flagged decision logging missing entirely. Both compound the three-failure-domain problem. The readiness probe must check DB connectivity; metrics must include DB query p99; the Cedar `reason` field (`rust-pdp-service-architecture.md:56-59`) must be logged at the plugin layer for per-request on-call diagnosis.
- **Entity trust boundary is aspirational prose (reinforced by Cantrill, Fowler, Schneier)**: The `entities` array is unconditionally caller-supplied (`rust-pdp-service-architecture.md:43-61`). The warning at line 157 is a comment, not enforcement. Fowler surfaced that the AuthZen `subject.properties` field (`rust-pdp-service-architecture.md:70-74`) makes this worse -- a freeform bag with no schema enforcement that flows into the Cedar entity mapping layer. One sanitization contract, enforced at both ingress points (Cedar-native and AuthZen mapping), is the required fix.

**New observations**:
- **Cedar skip-on-error as forbid bypass (from Schneier)**: Missing attribute causing a forbid policy to error results in the forbid being silently skipped (`cedar-policy-language.md:50-51`), allowing a permit to fire. Per-request input validation against schema before evaluation is required. The validate-before-swap at reload (`rust-pdp-service-architecture.md:109-113`) does not cover this path.
- **AuthZen endpoint as trust boundary regression**: The recommendation to map AuthZen externally to Cedar internally (`rust-pdp-service-architecture.md:83-85`) creates an additional unsanitized ingress point. The mapping layer must be explicitly designed as a validation boundary, not just a translation layer.
- **Policy delivery mechanism unspecified**: File watcher hot-reload works for a single instance; multi-instance coordination across a pod fleet requires a named solution. This is gestured at (`rust-pdp-service-architecture.md:139-143`) without being specified. Sidecar topology is not viable until this is designed.

**All 6 tension points -- final positions**:

1. **Go plugin vs Lua plugin**: Neither is universally correct. Document the latency budget first. If total authz overhead needs to be under 2ms, Lua's elimination of IPC is worth the tooling pain. If p99 SLA is 50ms and total overhead is 4ms, Go's developer experience wins. The 25% throughput floor from any Go plugin (`kong-plugin-architecture.md:139`) must be explicitly accepted or rejected against a stated budget -- not deferred.

2. **HTTP vs gRPC**: HTTP is the right default. The latency difference (gRPC saves ~0.5-1ms per call per `kong-plugin-architecture.md:148-151`) is noise compared to entity DB lookup latency. HTTP is easier to debug, easier to load-balance, and works with standard tooling. Use gRPC only if profiling shows the HTTP serialization overhead is a measured bottleneck. Don't add gRPC complexity speculatively.

3. **Sidecar vs remote PDP**: Topology is secondary to the plugin-side decision cache requirement. With a 30-60s TTL cache in the plugin, most requests never cross the process boundary regardless of topology. Mandate the cache first. Then: sidecar if you want co-located simplicity and have a named policy delivery mechanism; remote if you want independent PDP scaling. Both are acceptable with the cache in place.

4. **Policy hot-reload (arc-swap vs cedar-local-agent push)**: arc-swap is the right in-process mechanism -- lock-free reads, atomic swap, validate-before-swap (`rust-pdp-service-architecture.md:109-113`). The gap is not the swap mechanism, it's the policy delivery mechanism for multi-instance deployments. File watcher works for one instance; 10 Kong pods need a named solution (S3 sync, git webhook, DB-backed). This must be specified before sidecar topology is viable.

5. **Entity resolution (JWT vs DB vs cached hierarchy)**: Hybrid by sensitivity. JWT-sourced roles for low-sensitivity decisions -- acceptable revocation latency, no DB dependency on hot path. DB-verified for elevated privileges or security-critical attributes -- authoritative, but adds DB as a required dependency. The architecture must name which attribute classes go which path. "PDP should NOT trust PEP-supplied entity data for security-critical attributes" (`rust-pdp-service-architecture.md:157`) is a comment, not a contract. The API must enforce the boundary.

6. **Fail-open vs fail-closed**: Fail-closed always. `FailOpen bool` at `kong-plugin-architecture.md:68` must be removed as a per-route runtime config toggle. The architecture's fail-closed default in the PDP (`rust-pdp-service-architecture.md:216-222`) is correct and must not be overridable at the plugin config layer. Emergency override requires a deployment change with mandatory audit logging and automatic expiry. This is unanimous across all panelists.

**Score changes**:
- Deployment complexity: 0.40 -> 0.50. Cache-as-decoupling reduces the sidecar lifecycle concern if the cache is in place.
- Day-2 operations story: 0.50 -> 0.45. Decision logging gap and unspecified policy delivery mechanism are both worse than my initial read. The on-call runbook is further from complete than I originally scored.
