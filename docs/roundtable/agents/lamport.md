### Leslie Lamport -- Distributed Systems, Formal Specification, Temporal Reasoning
> The architecture has no written invariants, which means it has no correctness claims -- only performance numbers and optimistic prose.

#### Findings

| # | Finding | Severity | File:Line | Detail |
|---|---------|----------|-----------|--------|
| 1 | arc-swap atomicity is sufficient for single-instance but undefined for multi-instance | P1 | `rust-pdp-service-architecture.md:109-116` | The `reload()` method does two sequential `store()` calls: `policy_set.store(...)` then `schema.store(...)`. Between these two stores, a concurrent reader can load the new `PolicySet` with the old `Schema`. This is a transient inconsistency window. For a single process it is likely harmless (Cedar evaluates against the policy set independently of schema validation), but the architecture document states nothing about this invariant -- it is unspecified. Any reader that validates a request against schema post-load will see a torn state. |
| 2 | No consistency invariant for multi-instance policy distribution | P0 | `rust-pdp-service-architecture.md:138-143` | The policy store patterns table lists database-backed, S3, and Git options for multi-instance coordination, but specifies no invariant. Two PDP instances can simultaneously hold different active policy sets during a rolling reload. A single user request routed through two hops (e.g., batch endpoint with fan-out) can receive Allow from one instance and Deny from the other. This is not a race condition -- it is an unspecified consistency model. The architecture must state: "at most one policy version active across all instances at any time" or "eventual consistency with bounded staleness T" and enforce it. |
| 3 | Revocation window for cached entity hierarchy is unbounded without a deny list | P0 | `rust-pdp-service-architecture.md:150-158`, `entitlement-translation.md:176-179` | The recommended entity resolution pattern caches roles and hierarchy in an `ArcSwap<Entities>` refreshed "on change." The entitlement-translation wiki acknowledges: "Revocation latency: Security-critical revocations need a real-time deny list checked in context." However, the PDP architecture does not implement this. The maximum unauthorized access window equals the TTL of the entity cache (30-60s per `rust-pdp-service-architecture.md:175`). For a compromised or terminated user this is a 30-60 second authorization bypass. The architecture identifies the risk but does not specify the mechanism. |
| 4 | Fail-open/fail-closed state transition has no liveness guarantee | P1 | `rust-pdp-service-architecture.md:215-220`, `kong-plugin-architecture.md:67-68` | The architecture specifies fail-closed as default. The Kong plugin accepts a `fail_open: bool` config. Neither document specifies the circuit-breaker state machine: what conditions trigger the transition from closed to open, what conditions trigger recovery, and whether the transition is monotone or oscillating. Without a specification, operators cannot reason about availability under partial failure. A PDP process that crashes and recovers repeatedly could cause alternating permit/deny decisions across requests if the circuit breaker trips and recovers on short windows. |
| 5 | Decision cache invalidation on policy reload is not atomic with the reload | P1 | `rust-pdp-service-architecture.md:176-180` | The invalidation strategy "Policy-change-triggered (invalidate all on reload)" is listed but not ordered relative to the reload itself. If cache invalidation happens before the new policy set is stored (pre-swap), requests served during the gap hit the DB and evaluate against the old policy. If invalidation happens after the swap (post-swap), stale cached decisions persist briefly. Neither ordering is wrong per se, but neither is specified. The correct invariant is: "no cached decision is valid under the new policy set." This requires either invalidating after swap completion or versioning cache entries against the policy set. |
| 6 | HTTP vs gRPC decision lacks a failure-mode specification | P2 | `kong-plugin-architecture.md:147-148` | The latency delta between HTTP (~2-4ms) and gRPC (~1-3ms) is documented. Missing: what happens when the connection drops mid-request on each protocol. HTTP/1.1 over localhost will reset; gRPC streams may buffer. Neither is clearly better here, but the choice should be made based on failure semantics, not latency alone. |

#### Assessment

The architecture documents are well-researched on implementation mechanics but systematically avoid stating invariants. A formal specification -- even a brief TLA+ sketch or an informal invariant table -- would expose the gaps immediately. The documents read as "here is how to build it" rather than "here is what it must guarantee."

The most serious gap is the multi-instance policy distribution problem. The architecture lists file-based, database-backed, S3, and Git patterns but specifies no consistency model for any of them. Two concurrent requests during a rolling policy update can receive contradictory decisions. This is not a hypothetical: any zero-downtime deployment that updates policies on multiple PDP instances will produce this window. The duration is bounded by the time between the first instance reload and the last, which under Kubernetes rolling updates can be tens of seconds. The architecture must either (a) specify a global barrier on policy activation, (b) adopt a epoch-based versioning scheme where decisions carry the policy epoch, or (c) explicitly declare eventual consistency with a stated maximum staleness bound.

The revocation window in entity caching is the second serious gap. The entitlement wiki correctly identifies the problem and suggests a real-time deny list (`entitlement-translation.md:176-179`), but the PDP architecture does not implement one. The cache TTL is the security boundary for role revocation. At 30-60 seconds, a terminated employee retains authorization for up to one minute. Whether this is acceptable depends entirely on the threat model, which the architecture does not state. The architecture should either (a) implement the deny list, (b) lower the TTL for security-critical entities, or (c) explicitly document the accepted revocation latency as a design constraint.

The arc-swap hot-reload is nearly correct. The validation-before-swap invariant (`rust-pdp-service-architecture.md:133`) is the right approach and correctly prevents broken policies from becoming active. The residual issue is the two-store sequence: `policy_set.store()` followed by `schema.store()`. These are not atomic relative to readers. A reader between the two stores sees a new policy set with an old schema. Whether Cedar's runtime behavior is well-defined in this state is not documented. The fix is either to wrap both fields in a single `Arc<(PolicySet, Schema)>` and do a single swap, or to document that Cedar evaluation does not cross-reference schema at evaluation time (making the torn state harmless).

Before this ships, I want to see: (1) an explicit consistency model for multi-instance policy distribution with a stated maximum divergence window, (2) a revocation deny list or an explicit documented acceptance of the TTL window as the security boundary, and (3) the two-store arc-swap race condition resolved or proven harmless with a citation to Cedar's evaluation semantics.

#### Deliberation Addendum

**Positions revised**:
- Finding 1 (arc-swap torn-read, P1 -> P2): DHH correctly pointed out that Cedar's `Authorizer::is_authorized()` takes `(&Request, &PolicySet, &Entities)` with no Schema argument -- the schema is not referenced at evaluation time, only at validation time in `reload()`. The torn-read window between the two `store()` calls does not affect concurrent evaluations. Downgraded to P2: the structural inconsistency window exists and future readers of both fields must be audited, but the immediate evaluation path is safe. The single-Arc fix remains the correct structural remedy. The Assessment paragraph 3 overstated the risk; DHH's challenge is valid.
- Finding 3 (revocation window, framing adjusted): Schneier challenged whether this is a design gap or an implementation gap. Maintained as P0 but with revised framing: the specification gap leaves the invariant unenforced. The entitlement wiki names the solution category ("deny list") but does not specify the mechanism, ownership, latency bound, or failure behavior. Naming a problem is not the same as specifying a solution.

**Positions reinforced**:
- Finding 2 (multi-instance consistency, P0): Unchallenged. Cantrill asked for the proposed invariant; provided epoch-versioning as the MVP approach and activation barrier as the strong-consistency alternative. Epoch versioning with drain-and-reload for security-critical changes is the correct recommendation. No panelist disputed the gap.
- Finding 4 (fail-open, P1, effectively P0): Cantrill ("trapdoor, not a circuit breaker") and Schneier ("skip authorization, not a trade-off") both sharpened this finding. The `FailOpen bool` field should not exist; replace with a circuit breaker with explicit state transitions, mandatory TTL on any open state, and alerting on state change. Fowler challenged this as operational misuse; Cantrill and Schneier challenged it as a structural security violation. Both are correct and compounding.
- Finding 5 (cache invalidation ordering, P1): Reinforced by Schneier (S2: no policy version component in cache key) and Antirez (cache sized for single-node, evicts under fan-out). These three gaps compound: unspecified invalidation ordering + no version component + fixed LRU eviction under load means authorization behavior is indeterminate under production fan-out.
- Go plugin over Lua (Tension 1): Cantrill challenged on latency grounds. Held: the IPC cost buys testable, type-checked JWT parsing. This is the confused-deputy attack surface identified by Schneier and Fowler. Correctness before performance for a security component. Correct sequence: Go first (correctness baseline), Lua port only if latency budget requires it.

**New observations**:
- From Schneier (S2) + Antirez: decision cache key `principal:action:resource` has no policy version component. Under high fan-out the fixed LRU (10,000 entries) evicts before TTL, meaning the cache provides no consistency guarantee at production scale. Must either (a) version the cache key against a policy epoch, (b) invalidate entire cache on reload, or (c) explicitly document that cache correctness is not guaranteed. Option (c) is acceptable for MVP only if stated explicitly.
- From Cantrill (C3): startup ordering gap. Kong can accept traffic before the PDP has loaded its first policy set. The `/health/ready` probe exists but the architecture does not specify that Kong waits on PDP readiness before accepting traffic. This is a fail-open window at pod startup not present in my original analysis.
- From Cantrill/Fowler/Schneier convergence: the `/v1/is_authorized` API accepts `entities: [...]` unconditionally (rust-pdp-service-architecture.md:43-61). The PDP needs two entity input channels: trusted (PDP-resolved from authoritative sources) and untrusted (PEP-supplied, schema-constrained to explicitly enumerated non-security-critical attributes). Cedar's schema system supports this distinction structurally.

**Score changes**:
- arc-swap hot-reload correctness: 0.65 -> 0.72 (evaluation path is safe; Finding 1 downgrade to P2 justified by DHH's challenge)
- Multi-instance consistency: 0.40 -> 0.40 (unchanged; no new information, gap remains unaddressed in the architecture)
- Entity resolution correctness: 0.55 -> 0.45 (cache fan-out + version interaction compounds the revocation window gap; worse than originally assessed)
- Fail-closed strategy: 0.60 -> 0.50 (startup ordering gap adds a second fail-open window not in original analysis; the boolean trapdoor is structurally worse than I initially scored)

#### Supplementary Addendum (late challenges relayed by orchestrator)

**Schneier: skip-on-error as injection vector (new P1)**

Conceded as a finding not in my original analysis. Cedar's skip-on-error semantics (cedar-policy-language.md:50-52): a forbid policy that errors during evaluation is skipped, not treated as deny. If a JWT omits a claim referenced in a forbid condition, the forbid silently skips and a permit fires. With attacker-controlled tokens this is deliberate: strip the claim, bypass the forbid. Cedar's formal verification is sound relative to its specification -- skip-on-error is in the spec. My original scoring of Cedar's correctness conflated engine correctness with PDP integration correctness; these are separate claims.

Fix: PDP responsibility, not Cedar's. At policy load time, analyze the policy set to build a required-attribute manifest for all forbid conditions. Validate incoming entity completeness against that manifest before evaluation. A missing required attribute on a forbid-referenced field must be treated as deny, not allowed to trigger skip. This is a PDP wrapper obligation.

Score: Cedar integration correctness (not previously scored) -- 0.50. Engine is sound; integration contract is unspecified.

**Cantrill: Arc<(PolicySet, Schema)> tuple swap sufficiency**

Single Arc<(PolicySet, Schema)> swap is necessary and sufficient for intra-process torn-read. arc-swap store() is sequentially consistent for the pointer; readers get the old pair or the new pair, never a mix. No stronger primitive needed for the single-process case. Multi-instance coordination requires a higher-level epoch agreement protocol -- that is a separate problem at a different abstraction level, not solved by the tuple swap.

**Fowler: sidecar vs remote, network partition as expected failure mode**

Fowler's upgrade of fail-open to P0 for remote deployment is correct and reinforces the sidecar recommendation. On localhost, partition = process crash; failure is binary and deterministic; fail-closed is straightforward to enforce. Remote deployment makes failure probabilistic (partial partition, timeout, retry inconsistency) -- exactly the operational context where FailOpen bool gets flipped "temporarily" under incident pressure. Sidecar is architecturally correct as the starting point. Remote deployment is viable only after exhaustive failure semantics specification: partition detection threshold, circuit-breaker state machine, mandatory alerting on state transition, and prohibition on per-route fail-open override.
