### AGI Accelerationist -- Systems Architect / Scale Visionary
> "Cedar's entity model is a bottleneck dressed as a feature: beautiful for human RBAC, fragile for agent delegation chains at scale."

#### Findings

| # | Finding | Severity | File:Line | Detail |
|---|---------|----------|-----------|--------|
| 1 | No delegated authority chain in entity model | P0 | `cedar-policy-language.md:58-63` | Cedar's `in` operator tests transitive DAG membership but the entity model has no concept of scoped delegation. Agent A spawning Agent B with a permission subset requires materializing a new entity with reduced parent set -- there is no native "delegate with attenuation" primitive. Each spawned agent needs its own entity pre-provisioned in the entity store. At 1000 concurrent agents spawning sub-agents, this becomes a write-heavy entity provisioning problem with no solution in the documented design. |
| 2 | Batch endpoint design is under-specified for agent workloads | P1 | `rust-pdp-service-architecture.md:88-90` | `POST /access/v1/evaluations` is mentioned with three execution semantics (`execute_all`, `deny_on_first_deny`, `permit_on_first_permit`) but the implementation uses `JoinSet::spawn_blocking` per request. At agent workloads (hundreds of resources per batch call), this spawns hundreds of blocking threads. The `spawn_blocking` pool in tokio is bounded (default 512). At 1000 agent requests/sec each with 50-resource batches, you saturate the blocking pool and queue depth explodes. No backpressure mechanism is documented. |
| 3 | Fail-closed under overload is asserted but not designed | P0 | `rust-pdp-service-architecture.md:214-222` | **Partially resolved (BL-163).** The Kong plugin now treats all non-200 PDP responses as 503+Retry-After rather than silently denying (ADR-006 compliance). The original "all agents denied at once" liveness catastrophe no longer applies at the plugin layer. **Remaining gap:** the PDP itself still has no admission control, queue depth limits, or circuit breaker. Under infrastructure saturation the PDP will queue requests until OOM or pool exhaustion -- but the plugin will surface this as 503+Retry-After to callers, not as silent false denials. The distinction between "security deny" and "infrastructure unavailable" is now visible to clients. |
| 4 | Sidecar recommendation creates N copies of entity cache with no coherence | P1 | `kong-plugin-architecture.md:154-168` | Sidecar deployment is recommended for latency. But sidecar means one PDP instance per Kong node. **Update (BL-167/168):** Plugin-side decision caching (ADR-003) and arc-swap policy cache wrapper (ADR-004) are now implemented. Decision cache uses `principal|action|resource` keys with configurable TTL (default 30s), 10K entry capacity, and explicitly excludes errors from caching. However, the cache key contains no policy epoch or version -- a policy reload does not invalidate cached decisions until TTL expiry. Cross-sidecar invalidation remains unaddressed: each sidecar expires independently on TTL. For agent identity, role revocations still have a staleness window bounded by cache TTL. Stampede behavior (synchronized TTL expiry under load) and cache hit ratio effectiveness are pending benchmark validation. |
| 5 | JWT claims for workload identity are shallow | P1 | `entitlement-translation.md:156-163` | **Partially resolved (BL-162, BL-165).** BL-165 removed X-Consumer-ID header trust -- the plugin now uses only `kong.Client.GetConsumer()`, eliminating the client-spoofable identity path. BL-162 unified the Kong plugin and PDP entity namespace, so entity classification (ADR-005 Tier 1/2/3 attribute partitioning) now distinguishes principal types. **Remaining gap:** delegation metadata is still absent from JWT claims -- there is no mechanism to encode (a) which parent agent spawned this agent, (b) what permission subset was delegated, (c) delegation depth/chain. Policy authors still cannot express attenuation constraints. The entity model differentiates principal types but not delegation provenance. |
| 6 | Go plugin vs Lua: wrong question, IPC is the wrong layer | P2 | `kong-plugin-architecture.md:173-181` | The decision matrix scores Go+HTTP as the starting recommendation. At agent scale, the 25% throughput reduction from external plugin IPC (`kong-plugin-architecture.md:137-144`) is not a "floor cost" -- it is a fixed tax on every authorization call that scales linearly. Lua eliminates IPC but the real bottleneck is the HTTP call from plugin to PDP (1-3ms per `kong-plugin-architecture.md:147-151`). At 1000 req/sec, that's 1-3 seconds of aggregate PDP latency per second -- sustainable only with decision caching. The question is not Go vs Lua; it's cache hit ratio. Without documented cache hit rate targets for agent workloads, the performance model is incomplete. |

#### Assessment

Cedar's formal guarantees (termination, Lean proofs, sub-ms evaluation) are real strengths. For the use case of authorizing human API requests via Kong, this design is sound. The PARC model maps cleanly to HTTP request context, the arc-swap hot-reload is correctly designed (validate-before-swap at `rust-pdp-service-architecture.md:108-116`), and the AuthZen interop layer (`rust-pdp-service-architecture.md:63-85`) is the right call for engine-agnostic PEPs.

The agent-scale failure modes are structural, not incidental. Cedar's entity model is a DAG where principals inherit permissions through parent membership. This works for static organizational hierarchies. Agent organizations are not static: they are dynamic spawn trees where Agent A creates Agent B with a scoped subset of A's permissions, and B may create C with a further-scoped subset. Cedar has no native "permission attenuation on delegation" primitive. The `in` operator gives you transitive membership but not "membership with a capability mask." Every dynamic delegation requires pre-provisioning an entity in the store -- which means an external system managing entity lifecycle at the same rate agents spawn and terminate. This is the missing infrastructure piece, and the documents do not acknowledge it.

The batch evaluation design (`rust-pdp-service-architecture.md:269-278`) uses `spawn_blocking` because Cedar evaluation is CPU-bound. This is correct for single requests. For batch calls with O(100) sub-requests per agent call, the tokio blocking thread pool becomes the bottleneck. The architecture needs either: (a) a rayon parallel iterator for CPU-bound batch eval, keeping it off the blocking pool, or (b) explicit concurrency limits on batch size with backpressure signaling. Neither is present.

The fail-closed guarantee (`rust-pdp-service-architecture.md:214-222`) has been partially addressed by BL-163: the Kong plugin now treats all non-200 PDP responses as 503+Retry-After, distinguishing "PDP unavailable" from "policy deny" at the client layer. This resolves the original liveness catastrophe where infrastructure saturation was indistinguishable from security denials. The PDP itself still lacks admission control -- under saturation it will queue until OOM or pool exhaustion -- but clients now receive retryable 503s rather than false 403 denials.

Before this design ships for agent workloads, I want to see: (1) a concrete delegation model -- how does Agent B get scoped permissions from Agent A without human intervention in the entity store; (2) batch evaluation with rayon or explicit pool sizing; (3) cross-sidecar cache invalidation for security-sensitive revocations (plugin-side caching exists per BL-167/168 but lacks policy epoch in cache keys); (4) ~~admission control on the PDP so infrastructure saturation produces 503+retry-after rather than silent fail-closed~~ **partially resolved** -- BL-163 surfaces PDP errors as 503+Retry-After at the plugin layer, but the PDP itself still needs admission control to avoid saturation-induced OOM.

#### Deliberation Addendum

**Positions revised**:
- F4 (sidecar cache incoherence) -- refined after Antirez challenge. Antirez's cache key finding (`rust-pdp-service-architecture.md:170-172`) is more precise: the bug is not just cross-sidecar coherence but that the key `principal:action:resource` encodes no entity epoch. The incoherence exists even in single-sidecar deployments. My finding stands but Antirez's is the sharper formulation -- both point to the same root cause.
- F6 (Go vs Lua framing) -- partially conceded to DHH and Antirez. Both argued Go is the right permanent choice and the Lua migration path is not equivalent effort. Downgrading from "wrong question" to "underspecified recommendation." The question is still cache hit ratio, but the plugin language choice should commit to Go early.

**Positions reinforced**:
- F1 (no delegation attenuation primitive) -- unchallenged by any panelist. DHH's implicit "you don't need this" is a scope argument, not an architecture response. Reinforced by Schneier's JWT trust boundary finding: if agent tokens carry no delegation provenance, the PDP cannot enforce attenuation even if Cedar supported it natively.
- F2 (batch blocking pool saturation) -- unchallenged. No panelist offered a competing design for CPU-bound batch eval at agent workloads.
- F3 (fail-closed under saturation = liveness catastrophe) -- reinforced. Six panelists flagged the fail-open config field independently from different angles. My proposed fix (503+Retry-After for PDP unavailability, 403 for policy denies) gained support from Fowler and is the actionable resolution.

**New observations**:
- From Schneier F5: Cedar skip-on-error semantics (`cedar-policy-language.md:50-51`) create an attacker-exploitable path at agent scale. An agent that induces evaluation errors on forbid policies by supplying incomplete entity data can cause denies to be skipped while permits fire. At agent scale this is a systematic attack surface -- agents are more likely to send partial or unexpected entity data than human-proxied requests. Mitigation: strict schema validation on all entity attributes before Cedar evaluation begins.
- From Hashimoto F6: No decision audit log architecture. Cedar's `diagnostics().reason()` returns determining policy IDs but there is no persistence layer. For agent organizations, decision logs are essential for debugging unexpected denies and auditing agent behavior post-hoc.
- From Antirez F1: Cache stampede on TTL boundary (`rust-pdp-service-architecture.md:163-178`) is additive to my sidecar incoherence finding. No jitter, no stale-while-revalidate, no probabilistic early refresh on the moka cache. Periodic synchronized expiry of 10,000 entries is a predictable DB spike.

**Score changes**:
- Scalability: 0.40 -> 0.38. Antirez's cache key epoch finding adds a correctness dimension to the scalability problem I had scoped as coherence only. The stampede finding adds another independent failure mode under load. **Post-deliberation update:** BL-163 resolves the silent-deny catastrophe (F3) and BL-167/168 add plugin-side caching (F4 partial). These improve operational resilience but do not change the score -- batch saturation (F2), cache key versioning, and cross-sidecar invalidation remain open.
- Composability: 0.50 -> 0.45. Schneier's skip-on-error attack surface means policy composition has an exploitable gap that specifically affects agent workloads (agents are more likely than humans to send partial or crafted entity data).
- Autonomy: 0.35 -> unchanged. No deliberation challenged the core finding that agents are passive policy subjects with no self-service delegation or entity provisioning API.
