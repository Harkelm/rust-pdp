## Remaining opportunities (not blocking)

### Further test utility extraction (integration.rs, security.rs)

integration.rs and security.rs were not updated because their server starters
have intentionally different route sets:

- integration.rs: includes /v1/policy-info; start_claims_server omits
  /admin/reload; start_server_with_epoch_header has middleware
- security.rs: omits /admin/reload (intentional -- tests what happens when
  admin endpoints are absent)

Their inline request builders (admin_allow_request, viewer_deny_request
equivalents) could be replaced with common:: imports in a future pass.
This would save ~60 more lines but is lower priority since the main
duplication hotspot (stress + reload_resilience + concurrency) is already
consolidated.

### What NOT to do

- Don't extract the tempdir-based helpers (new_store, write_baseline)
  from reload_resilience.rs into common/ -- those are specific to that
  test file's concern (filesystem fault injection).
- Don't try to unify the router configurations across test files -- some
  tests intentionally omit certain routes (security.rs omits /admin/reload,
  diagnostics.rs omits batch endpoint). A parameterized builder would
  over-engineer this.
